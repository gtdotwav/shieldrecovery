3. CONFIGURAÇÃO DO WHATSAPP CLOUD API (META)
================================================================================

3.1 PRÉ-REQUISITOS:
  - Conta Meta Business
  - App registrado no Meta for Developers
  - WhatsApp Business API habilitada
  - Número verificado e aprovado

3.2 VARIÁVEIS DE AMBIENTE:
  WHATSAPP_ACCESS_TOKEN=EAA...     # Token de acesso permanente
  WHATSAPP_PHONE_NUMBER_ID=12345   # ID do número no Meta
  WHATSAPP_WEBHOOK_VERIFY_TOKEN=meu_token_secreto
  WHATSAPP_APP_SECRET=abc123       # App Secret para verificar assinatura

3.3 CONFIGURAÇÃO DO WEBHOOK NO META:

  URL do Webhook:
    https://pagrecovery.com/api/webhooks/whatsapp

  Verify Token:
    Valor configurado em WHATSAPP_WEBHOOK_VERIFY_TOKEN

  Campos subscritos:
    - messages (receber mensagens inbound)
    - message_status (rastrear delivery/read)

3.4 VERIFICAÇÃO DE ASSINATURA:
  O webhook verifica a assinatura usando x-hub-signature-256:

    Header: x-hub-signature-256: sha256=<hex_digest>
    Secret: WHATSAPP_APP_SECRET ou WHATSAPP_WEBHOOK_VERIFY_TOKEN
    Payload: raw body da requisição
    Método: HMAC-SHA256, comparação timing-safe (timingSafeEqual)

3.5 CHALLENGE (GET):
  GET /api/webhooks/whatsapp?hub.mode=subscribe
                            &hub.challenge=CHALLENGE_STRING
                            &hub.verify_token=MEU_TOKEN

  Resposta: retorna hub.challenge se verify_token confere

4. CONFIGURAÇÃO DO WHATSAPP WEB API (EVOLUTION)
================================================================================

4.1 PRÉ-REQUISITOS:
  - Instância Evolution API rodando (Docker/VPS)
  - Sessão WhatsApp Web criada (scan QR code)

4.2 VARIÁVEIS DE AMBIENTE / CONNECTION SETTINGS:
  whatsappProvider: "web_api"
  whatsappApiBaseUrl: "https://evolution.meuservidor.com"
  whatsappAccessToken: "minha_api_key"
  whatsappWebSessionId: "nome_da_sessao"

4.3 ENDPOINTS DA EVOLUTION API USADOS:

  Conexão:
    POST /instance/connect/{sessionId}     → Gera QR code
    GET  /instance/connectionState/{sid}   → Status da sessão
    POST /instance/logout/{sessionId}      → Desconectar

  Webhook:
    POST /webhook/set/{sessionId}
    Body: { webhook: { enabled: true, url: "...", events: [...] } }

  Envio:
    POST /message/sendText/{sessionId}
    Body: { number: "5511999999999", text: "Mensagem" }

4.4 ESTADOS DA SESSÃO:
  disconnected → pending_qr → connected → expired → error

  Armazenado em connection_settings:
    whatsapp_web_session_status
    whatsapp_web_session_qr_code (base64 data URL)
    whatsapp_web_session_phone
    whatsapp_web_session_error
    whatsapp_web_session_updated_at

4.5 FLUXO DE CONEXÃO:
  1. Admin acessa /connect no dashboard
  2. Sistema chama POST /instance/connect/{sessionId}
  3. QR code exibido na tela (whatsapp_web_session_qr_code)
  4. Usuário escaneia com WhatsApp do celular
  5. Status muda para "connected"
  6. Sistema configura webhook automaticamente

5.5 PROCESSAMENTO (MessagingService.handleWhatsAppWebhook):

  1. Detecta provider (cloud_api vs web_api) pelo formato do payload
  2. Extrai mensagens inbound (extractInboundMessages)
  3. Extrai atualizações de status (extractStatusUpdates)
  4. Para cada mensagem inbound:
     a. Busca lead pelo telefone (findLeadByContact)
     b. Upsert conversa (canal=whatsapp, contactValue=telefone)
     c. Armazena mensagem (direction=inbound, status=received)
     d. Detecta opt-out → processOptOut() se detectado
     e. Trigger resposta IA se modo autônomo ativo
  5. Para cada status update:
     a. Atualiza status da mensagem (sent/delivered/read/failed)
     b. Registra timestamps (deliveredAt, readAt)

5.6 DETECÇÃO DE INTENT (AI ORCHESTRATOR):
  O sistema classifica a intenção do cliente via regex patterns:

  "1", "pix", "quero pix"          → payment_method_pix
  "2", "cartao", "credito"         → payment_method_card
  "3", "boleto"                    → payment_method_boleto
  "ja paguei", "transferi"         → payment_confirmed
  "quero pagar", "vou pagar"       → payment_intent
  "depois", "amanha", "mais tarde" → needs_time
  "como funciona", "duvida"        → question
  "nao quero", "caro"              → objection
  "humano", "atendente"            → human_handoff
  "parar", "cancelar", "sair"      → opt_out

7. SISTEMA DE CONVERSAS (INBOX)
================================================================================

7.1 ESTRUTURA:
  Cada conversa é identificada por (channel + contactValue), ex:
    channel: "whatsapp", contactValue: "5511999999999"

  Status da conversa: open → pending → closed
    - Mensagem inbound reabre conversa fechada automaticamente
    - Operador/IA pode mudar status manualmente

7.2 TABELA: conversations
  id                UUID (PK)
  lead_record_id    UUID (FK → recovery_leads.id)
  lead_public_id    TEXT (ex: "lead_abc123")
  customer_id       UUID (FK → customers.id)
  customer_name     TEXT
  channel           TEXT ("whatsapp" | "email" | "sms" | "voice")
  contact_value     TEXT (telefone normalizado ou email)
  assigned_agent_id UUID (FK → agents.id)
  status            TEXT ("open" | "pending" | "closed")
  last_message_at   TIMESTAMPTZ
  created_at        TIMESTAMPTZ
  updated_at        TIMESTAMPTZ

  UNIQUE INDEX: (channel, contact_value)

7.3 TABELA: messages
  id                  UUID (PK)
  conversation_id     UUID (FK → conversations.id)
  lead_record_id      UUID
  lead_public_id      TEXT
  customer_id         UUID
  channel             TEXT
  direction           TEXT ("inbound" | "outbound")
  sender_name         TEXT
  sender_address      TEXT (telefone/email)
  content             TEXT
  provider_message_id TEXT UNIQUE (wamid.xxx / MSG_ID_xxx)
  status              TEXT ("queued"|"received"|"sent"|"delivered"|"read"|"failed")
  created_at          TIMESTAMPTZ
  delivered_at        TIMESTAMPTZ
  read_at             TIMESTAMPTZ
  error               TEXT
  metadata            JSONB

7.4 METADATA DA MENSAGEM (JSONB):

    "kind": "recovery_prompt" | "ai_draft" | "operator_note",
    "generatedBy": "workflow" | "ai" | "operator",
    "nextAction": "send_initial_message" | "ask_payment_method" | ...,
    "inboundIntent": "payment_method_pix" | "payment_confirmed" | ...,
    "selectedMethodType": "pix" | "card" | "boleto",
    "customerName": "João",
    "productName": "Curso X",
    "paymentValue": 14990,
    "retryLink": "https://checkout.../s/abc123",
    "paymentUrl": "https://...",
    "pixCode": "00020126580014br.gov.bcb.pix...",
    "pixQrCode": "data:image/png;base64,...",

7.5 INTERFACE DO INBOX (UI):
  Layout 3 colunas:
    Coluna 1: Fila de conversas (lista com badge de não-lidas)
    Coluna 2: Thread (mensagens + form de resposta + botão "Responder com IA")
    Coluna 3: Contexto do lead (nome, valor, produto, status, link ativo)

  Caminho: /inbox?conversationId={id}

7.6 ENDPOINTS DA INBOX:
  GET    /api/inbox                  → Lista conversas + unreadCount
  GET    /api/inbox/:id              → Conversa + mensagens + lead vinculado
  PATCH  /api/inbox/:id              → Atualizar conversa
  POST   /api/inbox/:id/reply        → Enviar resposta manual (operador)
  POST   /api/inbox/:id/ai-reply     → Gerar e enviar resposta IA
  PUT    /api/inbox/:id/status       → Mudar status (open/pending/closed)

7.7 CONTROLE DE ACESSO:
  - Admin: vê todas as conversas
  - Seller: vê apenas conversas dos leads atribuídos ao seu agentName
  - Market: vê conversas dos leads do seu escopo
  - Verificação: canRoleAccessAgent(role, assignedAgent, viewerAgentName)

8. GESTÃO DE NÚMEROS E NORMALIZAÇÃO
================================================================================

8.1 NORMALIZAÇÃO DE TELEFONE:

  Input                    → Output
  ────────────────────────────────────────
  "(11) 99999-9999"        → "5511999999999"
  "11999999999"            → "5511999999999"
  "+55 11 99999-9999"      → "5511999999999"
  "5511999999999"          → "5511999999999"
  "999999999@s.whatsapp.net" → "999999999"

  Função: normalizeBrazilPhone(value)
    1. Remove tudo que não é dígito
    2. Se não começa com "55", adiciona "55" na frente
    3. Resultado: sempre 12-13 dígitos (55 + DDD + número)

  Função: normalizeWhatsAppRemoteJid(value)
    1. Remove sufixo @s.whatsapp.net (ou @c.us, @g.us)

8.2 COMO O NÚMERO É USADO:
  - conversations.contact_value: telefone normalizado (sem formatação)
  - messages.sender_address: telefone normalizado
  - Busca de lead por telefone: findLeadByContact({ phone })
  - Match: comparação direta de strings após normalização

10. FLUXO COMPLETO: WEBHOOK → IA → MENSAGEM → RECUPERAÇÃO
================================================================================

  MINUTO 0:
    Gateway envia webhook de pagamento falho
    ├── Assinatura verificada (HMAC-SHA256)
    ├── Evento normalizado (payment_failed/refused/expired)
    ├── Customer e Payment upserted
    ├── Recovery Lead criado (NEW_RECOVERY)
    └── 5 jobs enfileirados:
        ├── lead-created (imediato)
        ├── whatsapp-initial (T+6min)
        ├── email-reminder (T+30min)
        ├── agent-task/ai_follow_up (T+24h)
        └── escalation-review (T+72h)

  MINUTO 6:
    Worker processa job "whatsapp-initial"
    ├── AI classifica lead (probabilidade, urgência)
    ├── Gera link de checkout (ou PIX direto)
    ├── Gera mensagem de recuperação (template ou OpenAI)
    ├── Cria conversa (canal=whatsapp)
    ├── Envia mensagem via WhatsApp Cloud API
    ├── Envia PIX como mensagem separada (se disponível)
    └── Status do lead: CONTACTING

  MINUTO 30:
    Worker processa job "email-reminder"
    ├── Envia email via SendGrid (mesmo conteúdo adaptado)
    └── Registra mensagem na conversa (canal=email)

  CLIENTE RESPONDE (ex: "1" = PIX):
    Webhook WhatsApp recebe mensagem inbound
    ├── processInboundMessage()
    ├── Intent detectado: payment_method_pix
    ├── Checkout session criada com método=PIX
    ├── AI gera resposta: "Perfeito! O link e o Pix estão aqui embaixo."
    ├── Mensagem enviada + PIX code separado
    └── Status do lead: WAITING_CUSTOMER

  CLIENTE PAGA:
    Webhook do gateway: payment_succeeded
    ├── Payment.status = "succeeded"
    ├── Lead.status = RECOVERED
    ├── Próxima mensagem inbound: "Já paguei"
    ├── AI detecta intent: payment_confirmed
    └── Resposta: "Pagamento confirmado! Muito obrigado pela compra."

11. AGENTE AUTÔNOMO DE RECUPERAÇÃO
================================================================================

  Arquivo: src/server/recovery/ai/autonomous-agent.ts
  Cron: GET /api/agent/orchestrate (a cada 5 minutos)
  Auth: Bearer CRON_SECRET
  Timeout: 50s (hard bail, Vercel max=55s)

  7 TAREFAS AUTÔNOMAS:

  1. Processar mensagens inbound não respondidas
     - Para cada lead ativo com inbound sem resposta
     - Classifica intent → gera resposta IA → envia

  2. Executar etapas de cadência pendentes
     - Verifica follow_up_cadences com scheduled_at <= now
     - Dispara mensagem ou voz conforme a etapa

  3. Agendar cadências faltantes
     - Leads sem cadência criada → cria etapas 4-11
     - Respeita horário comercial (9h-20h, fuso por DDD)
     - Pula finais de semana

  4. Processar transcrições de chamadas
     - Calls com transcript mas sem summary
     - Analisa via Claude: sentimento, compromissos, objeções

  5. Atualizar scores de recuperação
     - Recência, valor, status → metadata.scorePriority

  6. Escalar para humano
     - Leads 3+ dias, alto valor (>=R$50k), estagnados

  7. Fechar leads exauridos
     - Leads 10+ dias sem resposta → status: LOST

12. CADÊNCIA DE FOLLOW-UP (11 ETAPAS)
================================================================================

  Arquivo: src/server/recovery/ai/cadence-engine.ts

  ┌────────┬─────────────┬────────────────────────────────────────────────┐
  │ Etapa  │ Timing      │ Ação                                           │
  ├────────┼─────────────┼────────────────────────────────────────────────┤
  │ 1      │ T+0 (job)   │ Lead criado, webhook registrado                │
  │ 2      │ T+6min      │ WhatsApp inicial (mensagem de recuperação)      │
  │ 3      │ T+30min     │ Email reminder                                  │
  │ 4      │ T+4h        │ Follow-up contextual (WhatsApp)                 │
  │        │             │ Fallback: SMS após 1h se não entregue           │
  │ 5      │ T+12h       │ Voice call (alto valor) OU WhatsApp casual      │
  │ 6      │ T+48h       │ Métodos alternativos de pagamento               │
  │        │             │ Negociação elegível (desconto se habilitado)     │
  │ 7      │ T+72h       │ Oferta de desconto OU última chance             │
  │ 8      │ T+5 dias    │ Email re-engagement                             │
  │ 9      │ T+5.5 dias  │ SMS (se habilitado para o seller)               │
  │ 10     │ T+5.5 dias  │ Voice call para muito alto valor (>=R$1000)     │
  │ 11     │ T+7 dias    │ Soft close com porta aberta                     │
  └────────┴─────────────┴────────────────────────────────────────────────┘

  Regras:
  - Cada etapa é criada como registro em follow_up_cadences
  - Agente autônomo verifica etapas pendentes a cada 5 minutos
  - Se cliente respondeu recentemente → pula etapa
  - Se lead já RECOVERED ou LOST → pula todas restantes
  - Respeita horário comercial (9h-20h, timezone por DDD do telefone)
  - Não dispara em finais de semana

14. PIX — CÓDIGO COPIA E COLA
================================================================================

  REGRA FUNDAMENTAL:
  O código PIX copia-e-cola NUNCA é enviado dentro do texto da mensagem.
  É sempre enviado como uma MENSAGEM SEPARADA logo abaixo.

  Motivo: facilitar que o cliente copie o código inteiro sem erros.

  FLUXO:
  1. Sistema gera checkout session com método=PIX
  2. Checkout retorna: pixCode, pixQrCode, pixExpiresAt
  3. Mensagem principal é enviada (texto + label "Pix copia e cola 👇")
  4. Mensagem separada é enviada contendo APENAS o código PIX
  5. Ambas ficam registradas na tabela messages com metadata

  EXEMPLO DE CÓDIGO PIX:
  00020126580014br.gov.bcb.pix0136a1b2c3d4-e5f6-7890...

  METADATA ARMAZENADA:
  {
    "pixCode": "00020126580014br.gov.bcb.pix...",
    "pixQrCode": "data:image/png;base64,...",
    "pixExpiresAt": "2026-04-17T18:00:00Z"
  }

16. COMPLIANCE — OPT-OUT E FREQUÊNCIA (LGPD)
================================================================================

16.1 OPT-OUT:
  Arquivo: src/server/recovery/services/opt-out-service.ts

  KEYWORDS DE OPT-OUT:
    "parar", "sair", "cancelar", "remover", "descadastrar",
    "nao quero", "não quero", "pare", "stop", "unsubscribe",
    "nao me mande", "nao envie", "quero sair"

  DETECÇÃO:
    - Mensagens curtas (<=30 chars): qualquer keyword match
    - Mensagens longas: keyword no início ou match exato

  PROCESSAMENTO:
    1. Detecta keyword na mensagem inbound
    2. Cria registro em contact_opt_outs
    3. Lead marcado como LOST
    4. Mensagem de confirmação enviada:
       "Pronto! Você não receberá mais mensagens por este canal.
        Se mudar de ideia, é só nos enviar uma mensagem."

  TABELA: contact_opt_outs
    id, channel, contact_value, reason, opted_out_at,
    source ("inbound_keyword"|"admin_manual"|"api"|"legal_request"),
    seller_key, metadata

16.2 FREQUENCY CAPPING:
  Arquivo: src/server/recovery/services/frequency-service.ts

  LIMITES (por seller_admin_controls):
    max_contacts_per_lead_per_day   (padrão: sem limite explícito)
    max_contacts_per_lead_per_week  (padrão: sem limite explícito)

  VERIFICAÇÃO (canContactLead):
    1. Verifica opt-out (contact_opt_outs)
    2. Verifica frequência (contact_frequency_log)
    3. Se qualquer check falha → mensagem não enviada

  TABELA: contact_frequency_log
    id, contact_value, channel, direction, sent_at,
    message_id, call_id, seller_key

  Cada mensagem outbound registra um log de frequência.

21. COMO INTEGRAR EM OUTRO PROJETO
================================================================================

21.1 VIA SDK (@pagrecovery/sdk):

  Instalação:
    npm install @pagrecovery/sdk

  Configuração:
    import { PagRecovery } from '@pagrecovery/sdk';

    const pr = new PagRecovery({
      apiKey: 'sk_live_xxxxxxxx...',
      baseUrl: 'https://pagrecovery.com',  // ou sua instância
    });

  Uso:
    // Listar leads em recuperação
    const { leads } = await pr.leads.list({ status: 'CONTACTING' });

    // Ver conversa com mensagens
    const conv = await pr.inbox.get('conv_uuid');

    // Enviar resposta manual
    await pr.inbox.reply('conv_uuid', { message: 'Olá!' });

    // Gerar resposta IA
    await pr.inbox.aiReply('conv_uuid');

    // Iniciar chamada de voz
    await pr.calls.create({ toNumber: '+5511999999999' });

    // Dashboard
    const data = await pr.dashboard.get();

    // Analytics
    const analytics = await pr.analytics.recovery();

  Webhook verification:
    import { verifyWebhookSignature } from '@pagrecovery/sdk/webhooks';

    const isValid = verifyWebhookSignature({
      secret: process.env.WEBHOOK_SECRET,
      signature: req.headers['x-signature'],
      timestamp: req.headers['x-timestamp'],
      rawBody: rawBodyString,
    });

21.2 VIA API REST DIRETA:

  Auth:
    POST /api/auth/token
    Body: { "email": "...", "password": "..." }
    Response: { "token": "...", "role": "admin", "expiresIn": 604800 }

    OU usar API Key:
    Authorization: Bearer sk_live_xxxxxxxx...

  Enviar mensagem para lead:
    POST /api/inbox/{conversationId}/reply
    Authorization: Bearer sk_live_xxx
    Body: { "content": "Mensagem de recuperação..." }

  Receber webhooks de pagamento:
    Configure webhook no seu gateway apontando para:
    POST https://pagrecovery.com/api/webhooks/shield-gateway/{sellerKey}

    Headers obrigatórios:
      X-Webhook-Id: unique-id
      X-Timestamp: unix-timestamp-seconds
      X-Signature: hmac-sha256-hex

    Payload:
    {
      "event_type": "payment_failed",
      "payment": {
        "id": "pay_xxx",
        "amount": 14990,
        "currency": "BRL",
        "status": "failed",
        "failure_code": "card_declined",
        "customer": {
          "name": "João Silva",
          "email": "joao@email.com",
          "phone": "5511999999999"
        },
        "order_id": "order_123",
        "payment_method": "credit_card"
      }
    }

21.3 VIA WEBHOOK BIDIRECIONAL:

  Para integrar com outro sistema que precisa saber quando
  um pagamento é recuperado:

  1. Configure seller_admin_controls com checkout_url do sistema externo
  2. O Shield Recovery envia webhooks de evento para o CRM externo
     via crm_api_url configurado em connection_settings
  3. Use o SDK para polling de status ou configure webhook reverso

21.4 SELF-HOSTED (WHITE-LABEL):

  1. Clone o repositório
  2. Configure variáveis de ambiente (seção 18)
  3. Execute migration SQL (seção 19)
  4. Deploy na Vercel (ou qualquer host Node.js)
  5. Configure NEXT_PUBLIC_BRAND para sua marca
  6. Configure webhook do gateway de pagamento
  7. Configure WhatsApp Cloud API ou Evolution API
  8. Pronto — sistema autônomo de recuperação rodando
